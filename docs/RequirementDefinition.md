# 요구사항 정의서 (과제 3)

## 1. 문서 개요

### 1.1 목적

본 문서는 클라우드 파일 공유 시스템(미니 드라이브)의 요구사항을 기능적 요구사항, 비기능적 요구사항, 인터페이스 요구사항, 개발 요구사항으로 구분하여 정의한다.

### 1.2 범위

- 대상 시스템: 웹 기반 파일 공유 시스템
- 사용자 범위: 일반 사용자, 관리자
- 운영 규모: 약 200명 동시 사용 환경

### 1.3 관련 문서

- `docs/Requirement.md`
- `docs/Expectation.md`
- `docs/ProjectManagementPlan.md`

## 2. 요구사항 작성 원칙

본 문서는 아래 작성 원칙을 적용하여 요구사항을 정의한다.

| 항목 | 작성 원칙 |
| --- | --- |
| 식별성 | 모든 요구사항은 고유 ID(`REQ-FR-*`, `REQ-NFR-*`, `REQ-IF-*`, `REQ-DEV-*`)를 가진다. |
| 명확성 | 한 요구사항은 한 기능/한 품질 기준만 포함하며 모호한 표현을 사용하지 않는다. |
| 검증 가능성 | 각 요구사항은 테스트, 검토, 시연 중 하나 이상의 방법으로 검증 가능해야 한다. |
| 측정 가능성 | 비기능/품질 요구사항은 수치 기준(응답시간, 가용률, 보관기간 등)을 포함한다. |
| 구현 독립성 | 요구사항은 구현 방법이 아니라 시스템이 제공해야 할 결과를 기술한다. |
| 추적성 | 요구사항은 기능/품질/인터페이스/개발 항목으로 분류하여 변경 시 추적 가능해야 한다. |

## 3. 기능적 요구사항 (Functional Requirements)

| ID | 그룹/분류 | 상세 설명 | 우선순위 |
| --- | --- | --- | --- |
| REQ-FR-001 | 파일 업로드 | 사용자는 문서, 이미지, PDF 등 업무 파일을 시스템에 업로드할 수 있다. | 높음 |
| REQ-FR-002 | 파일 정보 저장 | 시스템은 파일명, 업로더, 업로드 시각, 파일 크기 정보를 저장한다. | 높음 |
| REQ-FR-003 | 파일 다운로드 | 사용자는 권한이 있는 파일을 언제든 다운로드할 수 있다. | 높음 |
| REQ-FR-004 | 폴더 생성/수정/삭제 | 사용자는 파일 정리를 위해 폴더를 생성, 이름 변경, 삭제할 수 있다. | 높음 |
| REQ-FR-005 | 다단계 폴더 구조 | 시스템은 하위 폴더를 포함한 계층형 폴더 구조를 지원한다. | 높음 |
| REQ-FR-006 | 삭제 파일 복구 | 삭제된 파일과 폴더는 삭제일로부터 30일간 보관되며, 보관 기간 내 복구할 수 있다. | 중간 |
| REQ-FR-007 | 사용자 간 파일 공유 | 사용자는 다른 사용자에게 파일 또는 폴더를 공유할 수 있다. | 높음 |
| REQ-FR-008 | 공유 권한 설정 | 공유 시 보기, 댓글, 수정 권한을 구분하여 설정할 수 있다. | 높음 |
| REQ-FR-009 | 외부 공유 링크 | 사용자는 외부 접근용 공유 링크를 만들고 1일, 7일, 30일 중 하나의 유효 기간을 설정할 수 있다. | 높음 |
| REQ-FR-010 | 파일명 검색 | 사용자는 파일 이름을 기준으로 원하는 파일을 검색할 수 있다. | 높음 |
| REQ-FR-011 | 조건 검색 | 사용자는 날짜, 업로더, 파일 형식 조건으로 파일을 검색할 수 있다. | 높음 |
| REQ-FR-012 | 파일 버전 저장 | 파일이 수정되면 시스템은 이전 버전을 자동으로 보관한다. | 높음 |
| REQ-FR-013 | 버전 조회/복원 | 사용자는 이전 버전을 조회, 다운로드, 복원할 수 있다. | 높음 |
| REQ-FR-014 | 사용자 로그인 | 사용자는 이메일과 비밀번호로 시스템에 로그인할 수 있다. | 높음 |
| REQ-FR-015 | 관리자 계정 관리 | 관리자는 사용자 계정과 사용자별 저장 공간을 관리할 수 있다. | 높음 |
| REQ-FR-016 | 사용자 권한 제한 | 시스템은 사용자 권한에 따라 접근 가능한 기능을 제한한다. | 높음 |

## 4. 비기능적 요구사항 (Non-Functional Requirements)

### 4.1 품질 요구사항

2장에서 식별한 품질 요소(Security, Availability, Performance, Reliability, Usability, Maintainability)를 기준으로 정의한다.

| 품질 요소 | ID | 요구사항 |
| --- | --- | --- |
| 보안성(Security) | REQ-NFR-SEC-001 | 시스템은 모든 웹/API 통신에 TLS 기반 암호화 전송을 적용해야 한다. |
| 보안성(Security) | REQ-NFR-SEC-002 | 시스템은 인증된 사용자만 접근 가능해야 하며 권한 없는 기능 호출을 차단해야 한다. |
| 가용성(Availability) | REQ-NFR-AVL-001 | 시스템은 월간 서비스 가용률 99.5% 이상을 목표로 운영되어야 한다. |
| 성능(Performance) | REQ-NFR-PER-001 | 시스템은 200명 규모 사용 환경에서 파일 검색 요청의 평균 응답시간을 2초 이내로 유지해야 한다. |
| 성능(Performance) | REQ-NFR-PER-002 | 시스템은 일반 파일 다운로드 시작 응답을 3초 이내에 반환해야 한다. |
| 신뢰성(Reliability) | REQ-NFR-REL-001 | 시스템은 장애 발생 시 최근 백업 기준으로 파일 복구가 가능해야 한다. |
| 사용성(Usability) | REQ-NFR-USA-001 | 시스템은 주요 시나리오(업로드, 다운로드, 공유, 검색)를 사용자 가이드 없이 수행 가능하도록 UI를 제공해야 한다. |
| 유지보수성(Maintainability) | REQ-NFR-MAI-001 | 시스템은 인증, 파일관리, 공유, 검색, 버전 기능을 모듈 단위로 분리한 구조를 유지해야 한다. |

### 4.2 일반 비기능 요구사항

| ID | 요구사항 |
| --- | --- |
| REQ-NFR-GEN-001 | 시스템 시각은 서버 표준시 기준으로 기록되어야 하며 파일 이력 로그에 생성/수정 시각을 남겨야 한다. |
| REQ-NFR-GEN-002 | 시스템은 주요 이벤트(로그인, 파일 업로드/삭제, 공유 설정 변경)를 감사 로그로 저장해야 한다. |
| REQ-NFR-GEN-003 | 시스템은 사용자 데이터 및 로그의 백업 주기와 보관 기간 정책을 설정값으로 관리할 수 있어야 한다. |

## 5. 인터페이스 요구사항 (Interface Requirements)

| ID | 인터페이스 구분 | 요구사항 |
| --- | --- | --- |
| REQ-IF-001 | 사용자 인터페이스(UI) | 시스템은 웹 브라우저 기반 UI를 제공해야 하며 파일 업로드/다운로드/공유/검색 화면을 포함해야 한다. |
| REQ-IF-002 | 사용자 인터페이스(UI) | 시스템은 데스크톱(최소 1280px) 환경에서 주요 기능 화면을 정상 표시해야 한다. |
| REQ-IF-003 | 사용자 인터페이스(UI) | 시스템은 모바일 환경에서 파일 목록 조회와 다운로드 기능을 제공해야 한다. |
| REQ-IF-004 | 시스템 인터페이스(API) | 시스템은 클라이언트-서버 통신에 HTTP 기반 API를 제공해야 하며 요청/응답 형식은 JSON을 사용해야 한다. |
| REQ-IF-005 | 시스템 인터페이스(API) | 시스템은 인증 토큰 기반 API 접근 통제를 제공해야 한다. |
| REQ-IF-006 | 데이터 인터페이스(DB) | 시스템은 파일 메타데이터(파일명, 경로, 소유자, 권한, 버전, 생성/수정 시각)를 저장할 수 있어야 한다. |
| REQ-IF-007 | 외부 인터페이스(링크 공유) | 시스템은 외부 공유 링크 생성, 1일/7일/30일 유효 기간 만료, 비활성화 기능을 제공해야 한다. |
